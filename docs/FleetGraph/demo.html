The Problem

Why FleetGraph Exists

Teams discover drift too late. By the time a manager notices stale issues, missed standups, and blocked approvals, the sprint is already slipping.

Without FleetGraph

  • Stale issues sit untouched for weeks
  • Missed standups go unnoticed until retro
  • Approval bottlenecks block downstream work silently
  • Scope drift creeps in undetected
  • Managers manually piece together status from 5 different views

With FleetGraph

  • Stale issues flagged after 7 days of inactivity
  • Missing standups surfaced same-day
  • Approval bottlenecks detected at 3-day threshold
  • Scope drift caught via content analysis (topic mismatch, dangerous patterns)
  • One contextual panel with evidence-backed signals

5 Drift Signals

Heuristic filters detect these signals deterministically before any LLM tokens are spent.

Safety Model

Human-in-the-Loop Guarantees

FleetGraph is designed to inform and recommend, never to act unilaterally.

Read Path (Always On)

  • Chat tools are GET-only (read Ship data)
  • Deterministic filters run before LLM
  • Sensitive keywords blocked (API keys, passwords)
  • Rate limited: 12 msg/min in production
  • Every run traced in LangSmith

Write Path (Gated)

  • All writes require explicit human approval
  • Approval expires after 72 hours
  • CAS guard: WHERE status = 'pending' (no double-execute)
  • Validation fails closed (reject on bad payload)
  • Full audit trail in fleetgraph_audit_log

FleetGraph is the layer that notices when execution starts to drift, tells the team why, and helps them correct course before the plan becomes fiction.
