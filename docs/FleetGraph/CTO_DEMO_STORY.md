# FleetGraph CTO Demo Story

FleetGraph exists to give Ship a stronger operating loop. Ship already knows how to store plans, issues, approvals, and sprint activity. The problem is that most teams still discover drift too late. By the time a manager pieces together stale work, delayed approvals, and missed accountability signals, the sprint has already started slipping. The north star for FleetGraph is to catch that change in direction early and help the team stay on course.

That is why FleetGraph is designed as an execution-drift agent instead of a generic assistant. It does not wait for someone to ask the perfect question, and it does not live on a disconnected chatbot page. It stays embedded inside the product, tied to the current issue, sprint, project, or workspace view. When the user opens it on demand, it begins with that context immediately. When the system runs proactively, it looks for signals that usually precede lost momentum: stale issues, missing standups, approval bottlenecks, and scope drift.

The architecture makes that product promise credible. FleetGraph runs on one LangGraph graph for both proactive and on-demand modes. It starts by fetching Ship context through the REST API, applies deterministic filters before it spends model tokens, and then reasons only when there is a meaningful candidate worth analyzing. From there, the graph takes visibly different paths. A healthy run exits cleanly. A risky run produces an alert. A consequential recommendation pauses at a human gate before anything is written back into Ship. LangSmith traces every run so the behavior is inspectable rather than hand-wavy.

The live demo should feel like a control system, not a chat trick. As you scroll `FLEETGRAPH.md`, the story is that FleetGraph protects the plan. It watches for regression, explains why a specific part of execution is bending away from the intended path, and gives the operator a grounded next move. When you switch into the app, the proof is that the same system works inside the real workflow. The alert is contextual. The chat is scoped. The recommendation is evidence-backed. The write path stays gated by a human.

For the current demo build, the North Star panel also has a deterministic proof trigger. The header bolt runs scoped analysis for the page you are on. The adjacent break-test button backdates a few real issues related to that project or issue context, seeds actionable FleetGraph approvals into the same North Star thread, and lights up the existing orange attention banner plus the Action Items modal. That lets Max show the full loop live: drift appears, the summary is readable in context, and approval writes back a real state or priority change through the same guarded path the production system uses.

That combination is the real product value. FleetGraph does not replace the team. It gives the team earlier signal, less hunting, and better control. It turns Ship from a place where work is recorded into a place where work can be kept aligned with the north star.

If you need a closing line, use this: FleetGraph is the layer that notices when execution starts to drift, tells the team why, and helps them correct course before the plan becomes fiction.
