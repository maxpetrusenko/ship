# FleetGraph Trace Links

This document tracks LangSmith trace URLs for FleetGraph's required demo scenarios. Each trace demonstrates a different graph branch path with real Ship data.

## How to generate traces

Set the following environment variables before starting the server:

```bash
export LANGCHAIN_TRACING_V2=true
export LANGCHAIN_API_KEY=<your-langsmith-key>
export LANGCHAIN_PROJECT=fleetgraph-mvp
export OPENAI_API_KEY=<your-openai-key>
export FLEETGRAPH_API_TOKEN=<workspace-token>
export FLEETGRAPH_WORKSPACE_ID=<workspace-uuid>
```

Start the server with `pnpm dev`. Proactive traces are generated automatically by the 4-minute sweep, which fans out to sprint-level runs, issue-level runs, and linked project rollups so wrong-ticket additions can surface upstream. On-demand traces are generated by calling the API endpoints.

## Required Traces (minimum 2 different paths)

### Trace 1: Clean proactive sweep

**Branch:** `clean`

**Scenario:** A healthy active sprint with recent standup coverage, no stale issues, and all approvals current. The heuristic filter finds zero candidates and the graph exits via `log_clean_run` without invoking the LLM.

**Seeded state:**
- Active sprint with all issues in `in_progress` or `done` state
- All issues updated within the last 24 hours
- No overdue accountability items
- No reopened issues

**Expected outcome:**
- Branch: `clean`
- Candidate count: 0
- No alerts generated
- Audit log entry with branch=clean, candidateCount=0
- No LLM invocation (cost: $0)

### Trace 2: Inform-only alert (missing standup or stale issue)

**Branch:** `inform_only`

**Scenario:** An active sprint where a team member has overdue standup accountability items. The heuristic filter detects the missing standup signal, the LLM reasons about the risk and produces an inform_only assessment, and the graph delivers an alert to the manager (sprint owner). Stale issue, issue-level scope drift, and linked project rollup traces are produced from the same scheduler sweep.

**Seeded state:**
- Active sprint with `owner_id` set to the manager user
- At least one team member with `missingStandup === true` in accountability signals
- Or: an in-progress issue with `lastActivityDays >= 3`

**Expected outcome:**
- Branch: `inform_only`
- Candidate count: >= 1
- Alert persisted to `fleetgraph_alerts` with severity `low` (standup) or `medium` (stale)
- WebSocket event broadcast to workspace
- LLM produces summary with cited evidence
- Audit log entry with branch=inform_only

**Trace URL:** See FLEETGRAPH.md Test Cases table for live links.

### Trace 3: Confirm-action with human gate

**Branch:** `confirm_action`

**Scenario:** A review approval has been pending for 3+ business days. The LLM recommends adding a comment to nudge the reviewer and routes through the confirm_action branch. A pending approval is created and the graph pauses at the human gate checkpoint.

**Seeded state:**
- Issue with a pending review accountability item
- `pendingApprovalDays >= 2` (ideally 3+ to trigger high severity)
- Identifiable reviewer (approver) from accountability data

**Expected outcome:**
- Branch: `confirm_action`
- Assessment includes `proposedAction` with `actionType: "add_comment"`
- Pending approval created in `fleetgraph_approvals` with 72h expiry
- Alert broadcast via WebSocket
- Graph pauses via `interrupt()` inside `human_gate` when a checkpointer is present
- Approval card rendered in UI with countdown timer

**Trace URL:** See FLEETGRAPH.md Test Cases table for live links.

### Trace 4: Error fallback

**Branch:** `error`

**Scenario:** The Ship API returns a 500 error during context fetching. The graph catches the error, sets the error branch, and routes to `error_fallback` for structured logging. No speculative alert is generated.

**Seeded state:**
- Misconfigured or unavailable Ship API endpoint
- Or: invalid `FLEETGRAPH_API_TOKEN` causing 401 responses

**Expected outcome:**
- Branch: `error`
- Error log with `failedNode: "fetch_core_context"` or `"fetch_parallel_signals"`
- `retryable: true`
- `followUpAction: "retry"`
- No alert generated (fail-safe behavior)
- Audit log entry with branch=error

**Trace URL:** See FLEETGRAPH.md Test Cases table for live links.

### Trace 5: Workspace scope chat

**Branch:** `inform_only`

**Scenario:** A user opens FleetGraph from a screen with no entity context, so the launcher falls back to workspace scope instead of pretending the view is a project. The graph analyzes workspace-level accountability and sprint health, then responds with a workspace-scoped summary.

**Seeded state:**
- Dashboard, settings, or search screen with no entity context
- Workspace has active sprints
- At least one overdue accountability item or manager follow-up item

**Expected outcome:**
- Branch: `inform_only`
- Request uses `entityType: "workspace"` and `entityId: workspaceId`
- Scope chip shows workspace label
- Chat response summarizes workspace-level triage signals
- No project passthrough in the request payload

**Trace URL:** See FLEETGRAPH.md Test Cases table for live links.

## Verification checklist

After generating traces, verify each one in the LangSmith dashboard:

- [ ] Trace 1 shows nodes 1-4 and 11, with no LLM call at node 5
- [ ] Trace 2 shows nodes 1-7, with LLM call at node 5 producing inform_only
- [ ] Trace 3 shows nodes 1-5 and 8-9, with LLM call producing confirm_action and proposedAction
- [ ] Trace 4 shows nodes 1-2 or 1-3 then 14, with structured error in state
- [ ] Trace 5 shows workspace scope entering chat without project aliasing
- [ ] All traces have valid `runId` and `traceId` metadata
- [ ] Token usage is captured in traces 2 and 3

## Sharing traces

1. Open the trace in LangSmith
2. Click "Share" to generate a public link
3. Replace the `[PENDING]` placeholder above with the shared URL
4. Update the corresponding row in `FLEETGRAPH.md` Test Cases table
